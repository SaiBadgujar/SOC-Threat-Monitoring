# Splunk Basics – DNS Log Analysis

## 1. Introduction

Domain Name System (DNS) logs provide valuable insights into network activity by recording domain lookup requests made by devices. Analyzing DNS traffic helps security analysts identify suspicious domains, unusual traffic patterns, and potential malware communication.

In this lab, Splunk is used to ingest and analyze **Zeek DNS logs** in JSON format. The objective is to extract useful information such as the most frequently queried domains, the most active source IP addresses, and the distribution of DNS query types.

---

## 2. Lab Objective

The main objectives of this lab are:

- Learn how to **ingest DNS logs into Splunk**
- Understand how **Zeek DNS logs are structured**
- Use **Splunk Search Processing Language (SPL)** to analyze DNS data
- Identify **frequently queried domains**
- Detect **top source IPs generating DNS requests**
- Analyze **different DNS query types**

## 4. Uploading DNS Logs into Splunk

Follow the steps below to ingest the DNS log file into Splunk.

### Step 1: Open Splunk Web Interface

Open your browser and access the **Splunk Web interface**.

---

### Step 2: Navigate to Data Input

Go to:


Settings → Add Data


This section allows you to upload and configure log files for indexing.

---

### Step 3: Upload the Log File

Select **Upload** and choose the DNS log file:


`dns.log`


This file contains DNS queries generated in **Zeek JSON log format**.

---

### Step 4: Configure Source Type

Set the source type to:


`json`


Alternatively, you can create a custom source type called:


`dns`


Using JSON helps Splunk automatically extract fields such as:

- query
- qtype
- id.orig_h
- id.resp_h

---

### Step 5: Configure Index

Choose the index where the data will be stored.

Options include:

- Default index: `main`
- Custom index: `dns_lab`

For this lab we use:

```
dns_lab
```

---

### Step 6: Complete the Upload

Finish the upload process and confirm that the logs have been successfully indexed.

You can verify by running:

```
index=dns_lab
```

If events appear, the data ingestion was successful.

## 5. Log Analysis Tasks

After uploading the logs, Splunk SPL queries are used to analyze DNS activity.

---

### Task 1: Identify the Most Frequently Queried Domain Names

### SPL Query

```spl
index=dns_lab sourcetype="json"
| stats count by query
| sort -count
```
Explanation

- index=dns_lab searches within the DNS log index.
- stats count by query counts how many times each domain name was queried.
- sort -count sorts the results in descending order.

Purpose

- This query helps analysts identify:
- Frequently accessed websites
- Popular external services
- Potential malicious domains contacted repeatedly
- Frequent queries to suspicious domains may indicate malware communication or DNS tunneling activity.

---

### Task 2: Find the Most Active User IPs Generating DNS Traffic

```SPL Query
index=dns_lab sourcetype="json"
| stats count by "id.orig_h"
| sort -count
```
Explanation

- id.orig_h represents the source IP address that initiated the DNS query.
- stats count by id.orig_h counts the number of queries generated by each IP address.
- sort -count displays the IPs with the highest activity.

Purpose

- This analysis helps identify:
- Hosts generating excessive DNS traffic
- Possible compromised systems
- Systems performing automated lookups or scanning
- A system generating unusually high DNS queries could indicate malware activity or automated scripts.

---

### Task 3: Breakdown of DNS Query Types

```SPL Query
index=dns_lab sourcetype="json"
| stats count by qtype
```
Explanation

- qtype represents the type of DNS query.
- stats count by qtype shows how many times each query type appears in the logs.

Common DNS Query Types

- Query Type	Description
- A	Maps a domain name to an IPv4 address
- AAAA	Maps a domain name to an IPv6 address
- CNAME	Domain alias pointing to another domain
- PTR	Reverse DNS lookup (IP to domain)

Purpose

- This helps analysts understand:
- Normal DNS traffic patterns
- The types of lookups occurring in the network
- Potential anomalies in DNS usage
For example, excessive PTR queries may indicate reverse lookup scans.
